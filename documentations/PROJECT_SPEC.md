## 1. Problem Statement

Enterprise organisations regularly handle PDF documents containing sensitive data: employee contracts with PAN/Aadhaar numbers, data dictionaries with AWS credentials, financial reports with salary figures, and internal memos with hate speech or threatening language.

Manual review is slow (hours per document), error-prone (humans miss patterns), and impossible to scale (hundreds of contracts at quarter-end). Compliance auditors need to answer: *"Does this document contain something that violates our data policy?"* — instantly, with evidence, and a risk rating.

**The PDF Compliance Scanner solves this**: an AI-powered pipeline that ingests any PDF, runs four parallel compliance checks, scores risk at every page, and delivers a downloadable audit-ready report — all in under 30 seconds.

## 2. Project Scope

The system has **two distinct scanning modes**, both sharing the same UI shell, AI backend, storage layer, and alert infrastructure:

| Mode | Input | What It Checks |
|------|-------|----------------|
| **PDF Document Scanner** | Uploaded PDF file | PII, confidential data, encoding violations, abuse/hate speech inside document text |
| **Data Source Scanner** | Connected database / cloud / warehouse | PII/PHI/PCI column classification, schema change detection, compliance drift tracking |

Both modes feed a shared **AI Copilot** (RAG-powered) and a shared **Telemetry & Analytics dashboard**.

## 4. PDF Document Scanning Pipeline

### 4.1 LangGraph DAG Structure

```
ingest
  ├──→ pii_check           (parallel)
  ├──→ confidentiality     (parallel)
  ├──→ encoding_check      (parallel)
  └──→ abuse_check         (parallel)
        ↓ (all 4 complete)
      aggregate
        ↓
      build_report
        ↓
       END
```

All 4 compliance nodes run in **true parallel** after ingestion. This is the key performance feature — a 20-page PDF does not wait for PII to finish before starting abuse detection.

### 4.2 Pipeline State Schema (`PipelineState`)

The typed state object that flows through every node:

| Field | Type | Description |
|-------|------|-------------|
| `pdf_path` | str | Temp file path of uploaded PDF |

| `pages_text` | List[PageText] | Per-page extracted text + encoding metadata |
| `pii_results` | List[Dict] | Flags from PII node |
| `confidential_results` | List[Dict] | Flags from confidentiality node |
| `encoding_results` | List[Dict] | Flags from encoding node |

### 4.3 Detection Modules (Pipeline Nodes)

#### 4.3.1 Ingest Node
- Uses **PyMuPDF (fitz)** to extract text page-by-page
- Detects encoding per page using **chardet**
- Counts embedded images per page
- Produces `PageText` structs with: `page_num`, `text`, `char_count`, `detected_encoding`, `encoding_confidence`, `image_count`

#### 4.3.2 PII Detector Node
**Dual-engine approach**: Regex (fast) + AI (semantic verification)

12 PII pattern types detected:

| Pattern | Example | Method |
|---------|---------|--------|
| Email | `john.doe@company.com` | Regex + AI verify |
| Indian Mobile | `+91-9876543210` | Regex |
| Aadhaar | `1234 5678 9012` | Regex |
| SSN (US) | `123-45-6789` | Regex |
| Credit Card | `4111 1111 1111 1111` | Regex + Luhn check |
| Passport | `A1234567` | Regex |
| PAN Card | `ABLDE1234F` | Regex |
| Date of Birth | `01/01/1990` | Regex |
| IP Address | `192.168.1.1` | Regex |
| Physical Address | Multi-line patterns | Regex |
| Full Name | Adjacent name tokens | AI |
| GPS Coordinates | `lat/long pairs` | Regex |

Each flag produces a `Flag` struct: `{type, value (truncated), context (50–150 chars), confidence (0.0–1.0), risk_level, position}`.

#### 4.3.3 Confidentiality Node
**15 credential pattern types**:

| Pattern | Risk Level | Example |
|---------|-----------|---------|
| AWS Access Key | CRITICAL | `AKIA...` |
| AWS Secret Key | CRITICAL | `wJalrXUtn...` |
| GitHub Token | CRITICAL | `ghp_...` |
| OpenAI API Key | CRITICAL | `sk-...` |
| Stripe Key | CRITICAL | `sk_live_...` |
| Slack Token | CRITICAL | `xoxb-...` |
| Password (plaintext) | CRITICAL | `password=hunter2` |
| Private Key (PEM) | CRITICAL | `-----BEGIN RSA...` |
| JWT Token | HIGH | `eyJ...` |
| Connection String | HIGH | `postgresql://...` |
| Salary/Compensation | HIGH | `₹95,00,000` |
| Merger/Acquisition keywords | HIGH | `Project TITAN` |
| Trade Secrets | HIGH | custom keywords from rules.json |
| Financial Data | HIGH | Q-on-Q revenue figures |
| Internal-only labels | MEDIUM | `CONFIDENTIAL`, `do not distribute` |

AI semantic check runs on any text segment matching broad patterns to reduce false positives.

#### 4.3.4 Encoding Guard Node
**6 encoding check types**:
1. **UTF-8 validation** — flag non-UTF-8 byte sequences
2. **Non-ASCII density** — flag pages where >12% of characters are non-ASCII (configurable threshold)
3. **Language compliance** — detect languages not in `allowed_languages` list (default: `["en"]`) using langdetect
4. **OCR corruption** — detect common OCR artefact patterns (repeated special characters, garbled Unicode)
5. **Encoding confidence** — flag pages where chardet confidence < 0.6
6. **Multi-script detection** — flag documents mixing multiple scripts (Latin + Devanagari, etc.)

All encoding checks are **rule-based** (no AI calls) — deterministic and fast.

#### 4.3.5 Abuse Detector Node
**3-layer detection** for zero-tolerance content:

- **Layer 1**: Keyword/phrase regex patterns (threat words, slurs, explicit content markers)
- **Layer 2**: Contextual pattern matching (sentence-level detection of harassment patterns)
- **Layer 3**: AI semantic classification for ambiguous cases

6 abuse categories:
1. Hate speech (race, religion, gender, sexuality)
2. Threats and violence incitement
3. Sexual/explicit content
4. Harassment and targeted abuse
5. Illegal content references
6. Terrorism/extremism (zero-tolerance)

Child safety and terrorism categories have **zero tolerance** — any match triggers CRITICAL regardless of confidence.

#### 4.3.6 Aggregator Node
- Merges results from all 4 parallel nodes
- Computes `PageResult` for each page: per-category risk levels + overall page risk
- Computes document-level `summary`: total_flags, total_pages, per-category counts, highest_risk
- Sets `processing_complete = True` to gate report generation

#### 4.3.7 Report Builder Node
Uses **ReportLab** to generate:
- **PDF Compliance Report**: Cover page, executive summary, per-page risk matrix heatmap, full redaction table with masked values, page-by-page breakdown
- **JSON Report**: Machine-readable full result (stored alongside PDF)

### 4.4 Confidence Scoring & Risk Levels

Every flag has a confidence score `0.0–1.0`:
- **Regex-only match**: 0.85–0.99 (high determinism)
- **AI-confirmed match**: 0.90–0.99
- **AI-only match**: 0.70–0.89 (semantic, without regex anchor)

Risk levels are assigned per flag type:
- **CRITICAL**: SSN, Passport, PAN, AWS keys, Passwords, PEM keys, terrorism/CSAM
- **HIGH**: Email, DOB, API keys, Salary, PHI, PCI
- **MEDIUM**: Addresses, IP addresses, Encoding issues, Internal labels
- **LOW**: Unconfirmed AI suggestions, non-ASCII density near threshold

Document overall risk = worst risk level found across all pages.

### 5.2 Data Source Agent Pipeline

When a user clicks **RUN COMPLIANCE SCAN** on a registered source, three stages run sequentially:

**Stage 1 — Metadata Agent (LangGraph)**
- Connects to the data source
- Extracts schema: all tables, all columns with data types, row counts
- Computes SHA-256 hash of the schema JSON
- Compares to the previous snapshot stored in SQLite:
  - **Same hash** → no structural change, checks for statistical anomalies only
  - **Different hash** → runs the Change Engine for structural diff
- Saves new snapshot to `metadata_snapshots` table

**Stage 2 — Compliance Agent (LangGraph)**
```
classify_columns → ai_enhance → compute_risk_summary → save_results
```
- `classify_columns`: runs 25+ regex patterns from the Rules Engine against every column name
- `ai_enhance`: sends unclassified columns to the AI to catch ambiguous cases (e.g., `user_demographics`, `financial_notes`), only accepts AI findings with confidence ≥ 0.70
- `compute_risk_summary`: computes numeric risk score and compliance %
- `save_results`: persists to `ds_scan_results` and updates `ds_scan_runs`

**Stage 3 — Knowledge Indexing (ChromaDB)**
- Converts each scan result to a natural-language document:
  ```
  Data source: Production Postgres
  Entity: users.ssn_hash
  Risk type: pii — PII_SSN
  Risk level: critical
  Evidence: Column name: ssn_hash | Type: VARCHAR | Method: name_pattern
  Recommendation: CRITICAL: Encrypt with AES-256; strict RBAC; audit all access.
  ```
- Upserts into ChromaDB collection `compliance_knowledge` with metadata: `source_id`, `scan_id`, `entity_name`, `check_type`, `risk_level`, `flag_category`
- These documents are immediately available for Copilot RAG retrieval

**Stage 4 — Risk Trend Update**
- Computes risk trend entry: `{risk_score, pii_count, confidential_count, total_violations, compliance_pct, trend_date}`
- Upserts into `risk_trends` table (one row per source per day)

**Stage 5 — Alert Check**
- Compares scan's `highest_risk` against all active `alert_configs`
- If threshold breached, dispatches notifications via configured channels

### 5.4 Change Engine — Schema Diff

The Change Engine compares two metadata snapshots and produces typed `change` events:

| Change Type | Entity | Severity |
|---|---|---|
| `new_table` | table | high |
| `dropped_table` | table | critical |
| `new_column` | column | high |
| `dropped_column` | column | high |
| `type_change` | column | medium |
| `row_count_spike` | table (>300% growth) | medium |
| `row_count_drop` | table (>80% drop with >1k rows) | high |

Each change is stored in `detected_changes` with: `change_id`, `source_id`, `snapshot_before`, `snapshot_after`, `change_type`, `entity_type`, `entity_name`, `change_detail`, `severity`, `detected_at`.

### 5.5 Risk Scoring Formula

```
risk_score = min(100, sum of severity_weights)
  where:
    critical = 25.0 per violation
    high     = 10.0 per violation
    medium   =  3.0 per violation
    low      =  1.0 per violation

compliance_pct = (1 - flagged_columns / total_columns) * 100
```

```

    ↓
```

## 9. AI Provider Architecture

### 9.1 Runtime-Switchable Provider

The `AI_PROVIDER` environment variable controls which backend is used. It is read **dynamically at call time** (not cached at module load), so changing it via the UI dropdown on the Upload page takes effect immediately for the current scan.

```
AI_PROVIDER=groq       → Groq Llama 3.3-70B (default, free, fast)
AI_PROVIDER=gemini     → Google Gemini 1.5 Flash (free tier)
AI_PROVIDER=anthropic  → Anthropic Claude Sonnet 4.5 (paid)
AI_PROVIDER=ollama     → Local Ollama (any model, air-gapped)
```

### 9.2 Universal `call_ai()` Function

Single abstraction over all providers:
- **Input**: `system_prompt: str`, `user_message: str`, `max_tokens: int`
- **Output**: `{"content": str, "tokens": int}`
- **Retry**: Tenacity with exponential backoff — up to 5 attempts, 2s to 60s wait
- **Tracing**: Every call traced to Langfuse (`@observe` decorator, `update_current_generation`)
- **Privacy**: Input/output content redacted in Langfuse traces (`<redacted>`) — only token counts and model name are logged

### 9.3 Langfuse Observability Integration

- Each full pipeline run is a **Langfuse Trace** (via `@observe(name="pdf-compliance-scan")`)
- Session ID = `upload_id` (unique per scan)
- Each `call_ai()` invocation is a **Langfuse Generation** (nested under the trace)
- Token usage reported per generation: `input`, `output`, `total`
- Tags: `["compliance-scan"]`
- Provider and model name attached as metadata
- `get_client().flush()` called after pipeline completes to ensure trace delivery

## 13. Non-Functional Requirements

### 13.1 Performance
- **PDF processing**: ~2–5 seconds per page (regex-fast + AI for ambiguous cases)
- **Data source scan**: ~3–10 seconds for typical schemas (100–500 columns)
- **Parallelism**: 4 compliance nodes run concurrently via LangGraph
- **RAG retrieval**: Sub-100ms (ChromaDB cosine similarity, local persistent store)
- **AI latency**: Groq Llama 3.3-70B averages 400–800ms per call (rate-limited free tier)

### 13.2 Reliability
- **Retry logic**: 5 attempts with exponential backoff (2s → 60s max) for all AI calls
- **Non-fatal failures**: Encoding guard, AI enhancement, RAG indexing, alert dispatch — all non-fatal; pipeline completes even if individual checks fail
- **Graceful degradation**: Copilot shows raw database context if AI provider unavailable
- **WAL mode SQLite**: Concurrent reads + writes without corruption

### 13.3 Scalability Considerations
- **Single-user local**: Current architecture
- **Multi-user production**: Each scan writes to isolated upload_id; SQLite WAL handles concurrent reads; swap SQLite → PostgreSQL / RDS for horizontal scale
- **ChromaDB**: Persistent local store; swap for Pinecone / Weaviate / Qdrant for distributed production

### 13.4 Observability
- **LLM tracing**: Every AI call traced in Langfuse with token counts, model name, latency, status
- **Scan audit trail**: Every scan logged in SQLite with full JSON result
- **Alert history**: All dispatched alerts logged in `alert_history`
- **Schema history**: All metadata snapshots stored with timestamps and hashes

## 14. Security Architecture & Tradeoffs

### 14.1 Implemented Security Controls

| Control | Implementation |
|---------|---------------|
| API key isolation | Keys in `.env` (git-ignored); never hardcoded |
| Secret field masking | Password/key fields use `type="password"` in Streamlit UI |
| AI privacy | Scan content sent to AI is `<redacted>` in Langfuse traces |
| Confidence thresholds | AI findings below 0.70 confidence are rejected |
| Input validation | PDF file type enforced, 50MB size limit |
| WAL journaling | SQLite concurrent access safety |

### 14.2 Security Tradeoffs

| Tradeoff | Decision | Rationale |
|----------|----------|-----------|
| **AI provider API calls** — document text leaves the machine | Accepted | Core requirement; mitigated by provider selection (Ollama = local/air-gapped option) |
| **SQLite vs. encrypted DB** | SQLite (unencrypted) | Local dev prototype; swap to encrypted RDS/Postgres for production |
| **Connector credentials stored in SQLite** | Plain JSON blob in `connection_config` | Prototype tradeoff; production should use AWS Secrets Manager or Vault |
| **No authentication layer** | No login wall | Scope: local tool for trusted team; add `streamlit-authenticator` for multi-user |
| **Langfuse cloud tracing** | Opted in | Token counts are non-sensitive; actual content is `<redacted>` |
| **ChromaDB local (no auth)** | Accepted | Local storage; no network exposure; production should add auth |

### 14.3 GDPR / Compliance Posture

- The tool **detects** PII but does not store the raw PII text — only truncated/masked versions in flags
- `context` field in `Flag` stores 50–150 chars of surrounding text (not the PII value itself)
- `value` field in `Flag` stores truncated matched text only
- Full text is stored only in the local PDF temp file (deleted after scan via `os.unlink`)
- Reports are stored locally in `reports/` — not transmitted anywhere

### 15.3 AWS Production Architecture (Enhancement A6)

```
PDF Upload (Streamlit UI — ECS Fargate port 8501)
  ↓ upload
S3 Bucket (PDF storage)
  ↓ S3 event notification
Lambda (trigger → push to SQS)
  ↓
SQS Queue (decouples upload from processing)
  ↓
LangGraph Pipeline (ECS Fargate — same container)
  ├─ Parse Agent (PyMuPDF)
  ├─ Compliance Agents (Bedrock Claude / Groq)
  └─ Report Builder (ReportLab)
  ↓
SQLite on EFS (findings + lineage — persists across Fargate restarts)
  ↓
Streamlit Dashboard (violation trends, token costs)

Secrets → AWS Secrets Manager
Logs    → CloudWatch (5GB free tier)
```

**Why EFS?** ECS Fargate containers have ephemeral disk. Without EFS, every container restart wipes the SQLite database and ChromaDB index.

### Sample Panel Q&A

| Question | Answer |
|----------|--------|
| "How do you know the AI didn't hallucinate a violation?" | Confidence scoring — AI findings below 0.70 are rejected. Regex findings have deterministic confidence 0.80–0.99. Every flag shows its match method (name_pattern / value_pattern / ai). |
| "What if Groq is down?" | Switch AI_PROVIDER to gemini, anthropic, or ollama via the UI dropdown. The pipeline retry logic handles transient failures with exponential backoff up to 5 attempts. |
| "Can you handle 200 PDFs at quarter-end?" | Architecture supports it: replace Streamlit upload with S3 trigger, add SQS queue, run pipeline as ECS task per PDF. LangGraph's async-compatible design handles this cleanly. |
| "What happens if a PDF has scanned images?" | PyMuPDF reports `image_count > 0` per page. The encoding guard flags it. OCR integration (e.g., Tesseract) is the natural next step. |
| "Show me the audit trail." | Every scan has a UUID (upload_id), timestamp, provider used, full JSON result, and token counts — all in SQLite. The Langfuse trace links every AI call back to that scan. |
| "Is document content sent to third parties?" | Only the text content sent to the AI provider for semantic checks. Langfuse traces content is `<redacted>` — only token counts and model names are logged. Ollama option keeps everything local. |
